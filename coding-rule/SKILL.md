# 编码准则

用于减少常见 LLM 编码失误的通用行为准则。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，可直接采用更轻量的处理方式。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。暴露权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，先问。
- 如果存在多种理解方式，一并呈现——不要默默选一个。
- 如果存在更简单的实现方案，说出来。必要时提出反对。
- 如果有不清楚的地方，停下来。说清哪里困惑，问一下。

## 2. 简单优先

**解决问题所需的最小代码。没有推测性内容。**

- 不要添加需求之外的功能。
- 不要为单次使用的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要处理不可能出现情况的错误处理。
- 如果能写成 50 行却写了 200 行，重写。

问自己："一个资深工程师会说这过度复杂吗？"如果会，简化。

## 3. 精确修改

**只改必须改的。清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你想用不同方式。
- 如果发现无关的死代码，提一下——不要删。

当你的修改产生了孤儿：
- 移除因你的修改而变得无用的 import/变量/函数。
- 不要删除原本就有的死代码，除非用户要求。

检验标准：每一行修改都应能追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证。**

将编码任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修 bug" → "写能重现 bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于需要编码的多步骤任务，先写一个短计划：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

强的成功标准让你能独立循环。弱的标准（"让它能工作"）需要不断澄清。

## 5. 注释服务于读者

**注释解释为什么，而不是做什么。**

- 不写"遍历数组"这类注释——代码本身已经表明。
- 写"为什么用这个算法"：业务原因、性能考量、已知限制。
- 不为每行代码加注释——用可读的变量名和函数名代替。
- 如果需要注释解释代码在做什么，先重写代码。

例外：复杂算法、非常规方案、临时 workaround 需要简短说明。
