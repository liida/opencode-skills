# 项目工作流

用于减少常见 LLM 编程失误的工作流准则，包含编码行为规范与项目任务流程。

## 编码行为准则

### 1. 编码前先思考

**不要假设。不要隐藏困惑。暴露权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，先问。
- 如果存在多种理解方式，一并呈现——不要默默选一个。
- 如果存在更简单的实现方案，说出来。必要时提出反对。
- 如果有不清楚的地方，停下来。说清哪里困惑，问一下。

### 2. 简单优先

**解决问题所需的最小代码。没有推测性内容。**

- 不要添加需求之外的功能。
- 不要为单次使用的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要处理不可能出现情况的错误处理。
- 如果能写成 50 行却写了 200 行，重写。

问自己："一个资深工程师会说这过度复杂吗？"如果会，简化。

### 3. 精确修改

**只改必须改的。清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你想用不同方式。
- 如果发现无关的死代码，提一下——不要删。

当你的修改产生了孤儿：
- 移除因你的修改而变得无用的 import/变量/函数。
- 不要删除原本就有的死代码，除非用户要求。

检验标准：每一行修改都应能追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环直到验证。**

将编码任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修 bug" → "写能重现 bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于需要编码的多步骤任务，先写一个短计划：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

强的成功标准让你能独立循环。弱的标准（"让它能工作"）需要不断澄清。

### 5. 注释服务于读者

**注释解释为什么，而不是做什么。**

- 不写"遍历数组"这类注释——代码本身已经表明。
- 写"为什么用这个算法"：业务原因、性能考量、已知限制。
- 不为每行代码加注释——用可读的变量名和函数名代替。
- 如果需要注释解释代码在做什么，先重写代码。

例外：复杂算法、非常规方案、临时 workaround 需要简短说明。

## 项目任务流程

### 目标

为仓库内的日常开发任务提供统一、轻量、可持续的工作流约定：
- 在任务开始前读取稳定上下文
- 在编码前先理解现有实现与可复用模式
- 在完成后进行本地验证
- 将长期有效的信息写回仓库记忆

### 强触发条件

当出现以下任一情况时，应优先加载本 skill：
- 开始一个新的仓库任务
- 开始一个新项目的开发
- 用户要求建立或维护项目记忆
- 用户要求遵循统一的项目级工作流
- 任务涉及多文件修改、已有模式复用、统一验证或知识沉淀
- 用户要求以后默认读取某些仓库文件

所有任务都应加载本 skill 并记录记忆。

### 基本原则

#### 默认语言
- 默认使用简体中文交流，除非用户明确要求其他语言。
- 代码标识符、API 名称、配置项名称遵循项目既有约定，不强行中文化。

#### 先理解，再动手
- 在执行任务前，先确认需求、上下文和成功标准。
- 如果存在关键歧义、关键缺失信息或多种合理路线，应先澄清，不要默默假设。
- 对简单明确任务可直接处理；对复杂任务应先拆分步骤。

### 连续执行规则

#### 默认连续推进
- 当信息充分、风险可控且无需用户做关键决策时，默认连续完成可连续完成的步骤，减少不必要中断。
- 对多步骤任务，优先一次性推进到一个自然的可验证节点，而不是每做一步都停下来询问。

#### 必须暂停确认的情况

遇到以下情况应暂停并请求确认：
- 需求存在关键歧义
- 存在多条合理路线且取舍明显不同
- 涉及高风险或不可逆操作
- 涉及权限边界、生产环境、删除或覆盖数据
- 需要用户提供外部信息、凭据或明确决策

### 上下文检索规则

#### 编码前先看现有实现

在修改前，应尽量确认：
- 仓库里是否已有相似实现
- 是否已有可复用模块、函数、组件或模式
- 当前目录或模块的命名风格和组织方式
- 这个改动应该如何验证

#### 检索深度按任务复杂度调整
- 小改动：做到"足够不误改"即可。
- 中大型改动：应系统了解相关模块、依赖与验证方式。
- 不要求固定步骤、固定数量或固定模板，重点是检索充分，而不是完成打卡动作。

#### 复用优先
- 优先复用官方方案、成熟依赖、仓库现有实现和既有模式。
- 只有在现有方案明显不适用时，才新增自定义实现。

### 验证规则

#### 改动后必须考虑验证
- 完成改动后，应进行与任务风险、范围和类型相匹配的验证。
- 优先执行当前环境下可立即完成的最小本地验证。
- 验证可以包括：
  - 单元测试
  - 集成或功能测试
  - 构建
  - 静态检查
  - 运行最小复现命令
  - 人工检查关键路径

#### 验证力度按风险调整
- 小改动：最小必要验证。
- 中等改动：至少覆盖主要受影响路径。
- 高风险改动：应尽量补足更完整验证。

#### 外部验证可以使用，但不能替代基础判断
- 可以使用 CI、远程流水线或人工评审作为补充验证。
- 但不应跳过当前上下文中本可完成的基础分析和基本验证。

#### 如实说明未验证部分
- 如果某部分无法验证，应明确说明：
  - 为什么无法验证
  - 当前风险是什么
  - 建议的补充验证方式是什么
- 不应把"未验证"表述成"已通过"。

### 审查与质量自检规则

#### 复杂任务应做自检

对复杂、跨模块或高风险任务，输出前应至少自检以下维度：
- 需求是否满足
- 改动是否正确
- 是否引入明显重复实现
- 是否保持可维护性
- 是否存在明显风险或遗漏验证

#### 不强制量化评分
- 可以使用质量维度做自检。
- 不要求固定打分、固定阈值或形式化评分表。

### 安全规则

#### 敏感信息不得记录或泄露

不得输出、保存或建议提交以下内容：
- 密钥
- 令牌
- 密码
- 私钥
- 凭据
- 其他敏感配置

#### 高风险操作要显式提示

涉及以下情况时，应明确提示风险并采取保守策略：
- 生产环境操作
- 权限提升
- 删除或覆盖数据
- 不可逆变更
- 可能引入明显安全风险的实现

#### 安全不应被无意义泛化
- 普通低风险任务不应被过度安全流程阻塞。
- 但一旦涉及敏感数据、权限、生产或漏洞风险，安全优先级应提升。

### 记忆与仓库协作规则

#### 任务前读取

优先读取：`.ai/memory.md` → `AGENTS.md` → 其他记忆文件。若不存在则跳过，通过上下文检索补充。

#### 任务后回写

所有任务都应更新记忆：
- 用户偏好、项目背景、架构决策、开发约定
- 新项目必须回写：背景、目录结构、依赖版本、启动方式
- 简单任务可简记：任务类型、完成状态、关键结果
- 验证通过后检查更新 `.ai/memory.md`、AGENTS.md 等文档

不应记录：敏感信息、日志、构建输出、临时草稿。

#### 记忆要求
- 简洁稳定，便于快速扫描
- 旧信息过时后优先替换

### 任务分解规则

#### 复杂任务应拆分

如果任务：
- 跨多个模块
- 涉及多阶段实现
- 需要多种验证
- 需要并行处理多个子问题

则应先拆分为更小步骤，再执行。

#### 以可验证节点为阶段边界

拆分时优先按"能验证、能收敛、能交付中间结果"的节点来分段，而不是机械按文件或按工具分段。

### 不应作为全局规则的内容

以下类型规则不应出现在全局 skill 中：
- 强制固定步数的检索清单
- 强制固定数量的充分性检查
- 无条件要求所有任务都使用同一种思考工具
- 无条件禁止 CI、远程验证或人工评审
- 惩罚式"懒惰检测"规则
- "禁止 MVP""必须全量功能"这类绝对化表达
- 与正常工程实践冲突的极端规则


## 推荐的 `.ai/` 结构

```
.ai/
├── memory.md
├── context-summary-<task>.md
└── verification-report-<task>.md
```

其中：
- memory.md 用于长期记忆
- context-summary-<task>.md 用于较复杂任务的上下文摘要
- verification-report-<task>.md 用于较复杂任务的验证记录
